# README для модуля ядра SimpleFS

## Команды для проверки второго исправления
Для дозаписи через режим append необходимо выполнить, например, один из следующих наборов команд:
```
printf 'append ' > /mnt/simplefs/file_0
printf 'test' >> /mnt/simplefs/file_0
cat /mnt/simplefs/file_0
```

```
echo "append " | sudo tee /mnt/simplefs/file_1
echo "test" | sudo tee -a /mnt/simplefs/file_1
cat /mnt/simplefs/file_1
```
Скриншот демонстрирующий работу исправлений (Скриншот_append_update.png) приведен в директории Скриншоты

## Команды для проверки исправлений
Для демонстрации ошибки монтирования при двух некорректных суперблоках необходимо размонтировать систему и испортить оба суперблока:
```
sudo dd if=/dev/urandom of=/dev/loop3 bs=512 count=1 seek=0 conv=notrunc
sudo dd if=/dev/urandom of=/dev/loop3 bs=512 count=1 seek=1024 conv=notrunc
sync
sudo blockdev --flushbufs /dev/loop3
```
После чего выполнить монтирование:
```
sudo mount -t simplefs /dev/loop3 /mnt/simplefs
```
В результате система выведет ошибку: mount: /mnt/simplefs: mount(2) system call failed: Structure needs cleaning.
В логах будет сообщение вида: [19675.293847] SimpleFS: both superblocks corrupted - refusing to mount

Для демонстрации работы erase выполнить команду:
```
sudo python3 ./SimpleFS_ctl.py erase /mnt/simplefs
```
Тогда команда
```
ls /mnt/simplefs | wc -l
```
До выполнения erase даст вывод, например 2046, после выполнения 0.

Для дозаписи через режим append необходимо выполнить команду:
```
echo "test" | sudo tee /mnt/simplefs/file_1
```
Скриншоты выполнения команд приведены в директории Скриншоты

## Задание 
Необходимо реализовать собственную ФС на базе существующего блочного устройства. ФС должна содержать 2 копии superblock (в начале диска и с определенным смещением в N секторов). Каждый оставшийся сектор должен соответствовать файлу (при инициализации фс все файлы созданы). Все файлы одинакового размера и могут занимать от 1 до M секторов.  Данную ФС должно быть можно смонтировать в /mnt. Необходимо реализовать регистрацию в VFS для отображения файлов в userspace. В файлы можно писать и читать из них, остальные операции не предусмотрены (переименование, перемещение, статистика и т.д.).

У  Superblock-а Обязательно должна быть копия. Сектор для размещения копии superblock-а задается параметрами модуля. Необходимо предусмотреть проверку целостности данных superblock-а (подсчёт хеш суммы).

Также необходимо сделать userspace программу (любой удобный язык программирования) для демонстрации корректности работы модуля:

Программа обходит весь диск и в каждый файл сначала пишет случайное число, а затем читает его. С помощью CLI программа позволяет вызывать IOCTL из задания ниже.

Аргументы модуля:
* Название диска на базе которого должна быть создана ФС
* Смещения N для superblock-ов (отдельно для первой и второй копии)
* Максимальная длина имени файла
* Максимальный размер файла в секторах (M)

IOCTL:
* Обнуление всех файлов
* Стереть FS
* Получить список всей метаинформации (хэши) из файлов
* Получить маппинг секторов для заданного файла (передается аргументом)

## Основные структуры модуля
### simplefs_disk_sb
Основная структура диска. Она в отличии от остальных in-memory структур физически хранится и накапливает данные о файловой системе, что позволяет после размонтирования системы при повторном монтировании восстановить сведения, например размер, число и расположение файлов.

Поля структуры:
* magic – метка для быстрой проверки на соответствие формата;
* file_size_sectors – число секторов, занимаемых одним файлом;
* max_name_len – максимальная длина имени файла (задается пользователем, поэтому не учитывает терминирующий ноль);
* num_files – число файлов в файловой системе;
* sb1_sector – дубль сектора первого суперблока для проверки (защита от некорректного смещения);
* sb2_sector – аналогичный дубль для второго суперблока;
* first_file_sector – начало области файлов (дублируется что бы не зависеть от внешних констант);
* total_sectors – общее число секторов устройства;
* hash – хэш структуры на основе предыдущих полей.

### simplefs_sb_info
In-memory копия основной структуры. Она хранит те же поля (кроме magic и hash), но в других форматах (конвертация происходит при монтировании), что позволяет ускорить работу с файловой системой. Так же она обеспечивает интеграцию с VFS, т.к. именно её адрес хранится в s_fs_info.

### simplefs_inode_info
Структура предназначенная для встраивания inode. Помимо его стандартных сведений, структура хранит поле file_index. В нем содержиттся информация об индексе конкретного файла в файловой системе.

### simplefs_hash_list
Используется для хранения заголовка IOCTL переменной длины.  Имеет следующие поля:
* count – общее число файлов в файловой системе;
* capacity – число файлов которое можно поместить в буфер;
* copied – число файлов помещенных в буфер;
* _pad – фиксированное смещение для выравнивания указателя на буфер;
* hashes_ptr – указатель на буфер.

### simplefs_file_hash
Сведения о конкретном файле из массива хешей. Используется в виде структуры, для уменьшения контакта между пространствами ядра и пользователя, поскольку вместо нескольких отдельных массивов данных набор сведений получается как упакованная структура. Содержит следующие поля:
* file_index – индекс конкретного файла в файловой системе;
* name – имя конкретного файла в файловой системе;
* hash – хэш конкретного файла в файловой системе.

### simplefs_sector_map
Структура предназначена для хранения сведений о физическом расположении файла на диске. Содержит следующие поля:
* file_index – индекс конкретного файла в файловой системе;
* name – имя конкретного файла в файловой системе;
* _pad – фиксированное смещение для выравнивания блоков first_sector и num_sectors;
* first_sector – LBA первого сектора;
* num_sectors – длина файла в секторах.

## Набор команд для демонстрации работоспособности модуля.
Сборка модуля:
```
make
```
Дальнейшие команды необходимо выполнять с привелегиями суперпользователя (sudo) или от рута.

Создание образа для монтирования:
```
dd if=/dev/zero of=/tmp/simplefs.img bs=1M count=8
losetup -f --show /tmp/simplefs.img
```
Важно использовать номер устройства, выведенный второй командой.
Загрузка модуля:
```
insmod ./SimpleFS.ko device_name=/dev/loop3 sb1_offset=0 sb2_offset=1024 max_name_len=32 max_file_size=8
```
Аргументы этой команды заданы в задании:
* device_name – Название диска на базе которого должна быть создана ФС;
* sb1_offset и sb2_offset – Смещения N для superblock-ов (отдельно для первой и второй копии);
* max_name_len – Максимальная длина имени файла;
* max_file_size – Максимальный размер файла в секторах (M).
Монтирование:
```
mkdir -p /mnt/simplefs
mount -t simplefs /dev/loop3 /mnt/simplefs
```
Для просмотра логов монтирования можно воспользоваться командой:
```
dmesg | tail -10
```
Просмотр статистики подключенной фс:
```
cat /proc/mounts | grep simplefs
```
Ручная запись данных в файл и чтение её:
```
echo "test" | tee /mnt/simplefs/file_0
cat /mnt/simplefs/file_0
```
Получение дампа файла:
```
xxd /mnt/simplefs/file_0 | head -2
```
Запись данных в файлы через userspace программу:
```
python3 SimpleFS_ctl.py test /mnt/simplefs
```
Просмотр хешей файлов через userspace программу:
```
python3 SimpleFS_ctl.py hashes /mnt/simplefs | head -10
```
Просмотр расположения файла через userspace программу:
```
python3 SimpleFS_ctl.py map /mnt/simplefs file_100
```
Обнуление файлов через userspace программу:
```
python3 SimpleFS_ctl.py zero /mnt/simplefs
```
Стирание файлов через userspace программу:
```
python3 SimpleFS_ctl.py erase /mnt/simplefs
```
Для демонстрации использования второго суперблока, необходимо размонтировать систему и испортить первый суперблок, например командой:
```
sudo dd if=/dev/urandom of=/dev/loop3 bs=512 count=1 conv=notrunc
```
После этого необходимо повторить команды монтирования и чтения логов, описанные выше.
